# Conrad Seoul 상품권 & 증서 관리 대시보드(Certificate Ledger) 사용자 매뉴얼

본 매뉴얼은 Conrad Seoul의 **상품권 및 서비스 증서(Gift & Service Certificate)**를 효율적이고 투명하게 관리하기 위해 구축된 대시보드의 기능별 사용 방법과 회계 처리 규칙을 설명합니다.

---

## 1. 핵심 회계 규칙 및 상태 정의

대시보드의 모든 데이터는 회계적 정합성을 유지하기 위해 **차액 0원 원칙**을 따릅니다. 

### 💡 핵심 수식: 차액 정산 공식
$$\text{차액 (Variance)} = \text{금액(A)} - \text{매출금액(B)} - \text{잡이익(C)} - \text{환불액(D)} = 0원$$
* **금액 (A)**: 상품권의 최초 판매 면가 (Face Value)
* **매출금액 (B)**: 업장 사용 등으로 발생한 실제 매출 (Outlet Posting Amount)
* **잡이익 (C)**: 미사용 만료 또는 환불 위약금 등으로 계상된 잡이익 (AR Posting Amount)
* **환불액 (D)**: 고객에게 실제 현금/카드로 반환된 금액 (Refund Amount)

> [!IMPORTANT]
> 최종 정산(사용 완료, 환불, 만료 처리 등)이 끝난 모든 증서는 반드시 위의 공식에 따라 **차액이 0원**이 되어야 올바르게 마감됩니다.

---

### 📌 증서 상태(Status) 정의
| 상태명 (UI 표시) | 설명 |
| :--- | :--- |
| **유효 (ACTIVE)** | 발행 완료 후 아직 사용, 취소, 만료되지 않고 유통 중인 정상 상태. |
| **사용 완료 (USED)** | 업장에서 사용되어 매출(B)로 정산이 완료된 상태. |
| **취소/환불 (VOID)** | 발행 오류로 파기(Misprint)되거나 판매 후 환불(Refund)되어 무효화된 상태. |
| **잡이익 전환 (EXPIRED_RECOGNIZED)** | 유효기간이 만료되어 미사용 금액이 100% 잡이익(C)으로 회계 마감된 상태. |
| **잡이익 환원 (GRACE_USED)** | 잡이익 마감(소멸) 처리된 서비스 증서를 소멸시효(5년) 전에 고객이 방문하여 소급 사용한 상태. |

## 2. 기능별 상세 사용법

### 1) 상품권 판매 및 신규 등록 (New Issue)
상품권이나 서비스 증서를 판매했을 때 시스템에 등록하는 프로세스입니다.

1. 좌측 메뉴에서 **증서 목록(Certificate List)**을 선택합니다.
2. 우측 상단의 **`+ 새 증서 발행 (New Certificate)`** 버튼을 클릭하여 입력 패널을 엽니다.
3. 상황에 맞는 발행 모드를 선택합니다.
   * **연번 여러 장 (Numbered batch)** `[추천]` : 동일 상품을 연속된 증서 번호로 대량 등록할 때 사용합니다.
     * **종류(Category)**와 **서비스 포함내역(Detail)**을 선택합니다.
     * **시작 증서번호**와 발행할 **수량**을 입력하면 우측에 예상 번호 대역이 실시간 표시됩니다.
     * **금액, 발행일, 결제수단, 비고(판매자)** 등을 입력한 후 **`생성 (Generate)`**을 클릭하면 하단 목록에 자동 생성됩니다.
   * **1장씩 발행 (One at a time)**: 수동으로 직접 1장씩 값을 타이핑하여 등록합니다.
4. 하단 목록에 생성된 내역의 증서번호, 만료일, 금액 등을 최종 확인합니다.
5. **`전체 등록 (Register All)`**을 클릭하여 데이터베이스에 저장합니다.

> [!TIP]
> * **유효기간(만료일) 자동 계산**: 발행일을 선택하면 종류에 맞춰 만료일이 자동 지정됩니다. (Service 증서는 1년, Gift 상품권은 5년)
> * **증서번호 자동 제안**: 번호 입력 칸에서 `Tab` 키를 누르면 시스템에 등록된 마지막 번호 다음 번호가 자동으로 제안 입력됩니다.

### 2) 상품권 사용 처리 (Redemption)
고객이 업장에 방문하여 증서를 사용했을 때 매출로 인식하는 과정입니다.

1. **증서 목록**에서 사용 처리할 증서번호를 검색합니다.
2. 해당 증서 좌측의 체크박스를 선택합니다 (여러 건 동시 선택 가능).
3. 상단 툴바에 나타나는 **`사용 처리 (Use)`** 버튼을 클릭합니다.
4. 사용 처리 팝업창에서 상세 정보를 입력합니다.
   * **정상 사용**: 오늘 날짜의 사용일과 면가와 동일한 **매출금액(B)**을 확인하고 처리합니다. (잡이익 C = 0원)
   * **만료 후 사용 (Late Use)**: 
     * 유효기간이 지난 서비스 증서(Service Certificate)를 사용 처리하는 경우, 시스템이 회계 규칙에 따라 **매출(B) 90% / 잡이익(C) 10%**로 정산 금액을 자동 분할 제안합니다.
     * 이 경우 **매출등록일(Misc Rev Posting Date)**을 필수로 입력해야 합니다.
   * **금액권 부분 사용 및 잔액 환급**:
     * 10만원 금액권 등을 일부만 쓰고 남은 잔액을 현금 환급하는 경우, 실제 사용한 금액을 **매출금액(B)**에 적고, 남은 잔액은 **환불액(D)**에 기입합니다.
     * (예: 10만 원권으로 6만 원 결제 후 잔액 환불 $\rightarrow$ 매출 B: 60,000원, 환불액 D: 40,000원 입력)
     * 시스템 하단에 환급 법정 대상 비율 충족 여부(Conrad Seoul 규정: 60% 이상 사용 시 환급 가능)와 차액 정합성이 실시간으로 계산 및 표시됩니다.
5. 영수증 번호나 객실 번호 등의 식별 값을 **비고(Bill No.)**란에 기입합니다.
6. **`일괄 처리 / 사용 처리`** 버튼을 눌러 완료합니다.

### 3) 취소 및 환불 처리 (Void & Refund)
발행 오류로 증서를 파기하거나 판매된 증서를 반품/환불할 때 사용합니다.

1. **증서 목록**에서 ACTIVE 상태인 증서를 검색하여 선택합니다.
2. 상단 툴바에서 **`취소/환불 (Void/Refund)`** 버튼을 클릭합니다.
3. 환불 유형을 선택합니다.
   * **미판매 취소 (Misprint)**: 잘못 인쇄되거나 발행 도중 실수가 발생하여 고객에게 전달되지 않고 즉시 폐기하는 경우입니다. 금액 이동이 없으므로 환불액은 0원 처리됩니다.
   * **판매 후 환불 (Refund)**: 이미 판매된 증서를 환불 조치하는 경우입니다.
     * **위약금 없이 전액 환불**: 면가 전체가 **환불액(D)**으로 계상되어 환불 처리됩니다.
     * **위약금 10% 적용**: 회계 규정에 따라 면가의 10%를 위약금 명목의 **잡이익(C)**으로 귀속시키고, 나머지 90%만 고객에게 **환불액(D)**으로 지급 정산합니다. (팝업창에서 미리보기 창을 통해 정산 금액을 최종 확인할 수 있습니다.)
4. 환불이 일어난 정확한 **환불일**을 선택하고 **`일괄 처리`**를 클릭합니다.

---

### 4) 유효기간 만료 및 소멸 인식 (Expiry Queue & Write-Off)
유효기간이 경과한 증서들을 잡이익 계정으로 일괄 마감(Write-off) 처리하는 작업입니다. 주로 분기/연말 회계 마감 시 수행합니다.

1. 좌측 메뉴에서 **만료 대기열 (Expiry Queue)**로 이동합니다.
2. 화면 상단에 현재 기준일(As of Date) 기준으로 만료된 건들이 대기열 목록에 노출됩니다.
   * **FORECAST 현황판**: 화면 상단의 표는 올해 누적 완료된 전환 건수, 현재 대기 중인 건수 및 향후 6개월 단위 만료 예정인 유효 증서들의 수량과 금액을 카테고리별로 예측하여 보여줍니다.
3. 마감 처리 방식을 선택합니다.
   * **선택 항목 만료 처리**: 특정 증서들만 체크하여 **`선택 항목 인식 처리`**를 클릭합니다.
   * **연말 일괄 만료 처리**: 대기열의 전체 건을 한 번에 마감하려면 **`전체 연말 일괄 인식 처리 (Year-End)`**를 클릭합니다.
4. 마감이 완료되면 해당 증서들은 100% **잡이익(C)** 계정으로 금액이 이관되며, 상태는 **`잡이익 전환 (EXPIRED_RECOGNIZED)`**으로 변경됩니다.

---

### 5) 잡이익 환원 처리 (Grace Use)
이미 유효기간이 지나 연말 잡이익으로 100% 마감 처리된 **서비스 증서(Service Certificate)**에 대해, 고객이 소멸시효(발행일로부터 5년) 이내에 찾아와 예외적으로 사용을 요청하는 경우 적용하는 소급 처리입니다.

1. **증서 목록**에서 해당 증서번호를 검색합니다. (상태가 **잡이익 전환** 상태여야 합니다.)
2. 해당 증서의 체크박스를 선택합니다.
3. 상단 툴바에서 **`잡이익 환원 (Grace Use)`** 버튼을 클릭합니다.
4. 팝업창에서 고객이 방문한 사용일을 지정하고 확인을 누릅니다.
5. 처리 완료 시:
   * 기존에 100% 잡이익(C)으로 묶여 있던 돈 중 **90%가 매출금액(B)으로 반제**되어 되돌아가고, **10%만 패널티 성격의 잡이익(C)**으로 원장에 남습니다.
   * **잡이익 원장(Misc Revenue Ledger)**에는 기존 전환 금액의 취소 반제 기록과 90% 환원 지급 기록이 자동으로 기입됩니다.
   * 증서 상태는 **`잡이익 환원 (GRACE_USED)`**으로 변경됩니다.

> [!WARNING]
> * **Gift 상품권**: Gift 상품권은 법적으로 Grace Use 예외 규정이 없으므로 만료 전환 완료 후 환원이 불가능합니다.
> * **5년 만료 경과**: 발행일로부터 5년이 완전히 지난 서비스 증서는 법적 소멸시효가 완성되어 잡이익이 최종 확정되므로 이 처리를 수행할 수 없습니다.

---

### 6) 유효기간 연장 (Extend Expiry)
부득이한 사유로 만료 전에 유효기간을 연장해주어야 할 때 사용하는 프로세스입니다.

1. **만료 대기열** 목록 혹은 증서 목록에서 대상 증서 우측의 **`유효기간 연장 (Extend)`** 버튼을 클릭합니다.
2. 유효기간 연장 입력 창에서 다음 정보를 작성합니다.
   * **새 만료일 (New Expiry Date)**: 오늘 이후의 미래 날짜를 지정합니다.
   * **Mate Approval # (관리자 승인번호)**: 매니저 승인 번호를 반드시 기입해야 합니다 (필수).
3. **`연장`**을 클릭합니다. 
4. 연장 완료 시 증서는 만료 대기열에서 사라지고 다시 **유효 (ACTIVE)** 상태로 돌아가 정상적으로 사용할 수 있게 됩니다.

---

### 7) 데이터 정합성 점검 (Integrity Check)
사용자의 실수로 입력된 회계 오류 데이터를 사전에 검출하는 상시 점검 기능입니다.

1. 좌측 메뉴에서 **정합성 점검 (Integrity Check)**으로 이동합니다.
2. 시스템이 내부 정합성 룰을 바탕으로 오류 데이터를 실시간 감지하여 보여줍니다.
   * 예: $A - B - C - D \neq 0$인 차액 오류 건
   * 사용 완료인데 사용일자가 누락된 건
   * 만료 마감인데 실제 매출(B)에 금액이 들어가 있는 회계 오분개 건 등
3. 각 오류 항목 우측의 **`수정 (Fix)`** 버튼을 클릭하면 해당 증서 정보가 즉시 잠금 해제되어 대화상자나 목록에서 바로 수정할 수 있습니다.

## 3. 유용한 부가 기능

* **작업 되돌리기 (Undo)**:
  * 대시보드 상단에는 항상 가장 마지막에 수행한 작업(발행, 사용 처리, 환불 등)에 대한 정보바가 노출됩니다.
  * 실수로 대량 발행했거나 오분개한 경우 상단의 **`되돌리기 (Undo)`** 버튼을 클릭하면 한 단계 이전 상태로 즉각 복구됩니다.
* **엑셀 업로드 및 내보내기 (Import / Export)**:
  * **가져오기/내보내기** 메뉴에서 기존 레거시 엑셀 장부를 업로드하여 시스템에 한 번에 이관할 수 있으며, 현재 데이터베이스 내역을 종류별 엑셀 파일로 백업 다운로드할 수 있습니다.
  * 데이터 업로드 도중 오류가 감지되면 해당 데이터들은 자동으로 **`검토 필요(Needs Review)`** 플래그가 지정되어 증서 목록에 강조 표시됩니다.
* **담당자 기록 (Audit Log)**:
  * 대시보드에서 행해지는 모든 수동 데이터 변경, 잠금 해제 수정, 일괄 마감 등의 작업은 **감사 로그(Audit Log)** 메뉴에 변경 전/후 스냅샷과 함께 영구 기록됩니다.
  * 데이터 수정 시에는 기록 보존을 위해 **변경 사유(Change Reason)**를 적도록 강제화되어 있어, 철저한 내부 통제 및 회계 감사를 보장합니다.
